### Disfluency Cleanup (RU) — Design Document

Дата: 2025‑10‑23

Версия: 0.1 (Draft)

Автор: Artemonim's Speech Kit

Сcope: офлайновая автоматическая зачистка дисфлюенций в русской речи: вздохи, звуки‑филлеры (“э‑э”, “мм”), слова‑паразиты (например, “ну”, “типа”, “короче”).

---

### 1) Цели и критерии

- Минимизировать ручную чистку аудио за счёт автоматического выявления и безопасного применения правок.
- Не ломать тайминги видео/субтитров: по умолчанию — недеструктивная обработка (приглушение), опционально — вырез с кросс‑фейдом.
- Управляемость: флаги, пороги, лимиты, режимы strict/normal/aggressive.
- Трассируемость: JSON‑cutlist, EDL для NLE, лог метрик, возможность отката.

---

### 2) Объекты обработки

- Вздохи/дыхание: короткие шумовые/сустейн‑интервалы вне речи, 200–1200 мс.
- Звуки‑филлеры: “э‑э”, “мм”, растянутые гласные без смысловой нагрузки.
- Слова‑паразиты (ru): “ну”, “типа”, “как бы”, “короче”, “в общем”, “по сути”, “получается”, “значит”, “это самое”, “вот”, “скажем так”, “так сказать”, “как сказать”, “да”, “в принципе”, “в итоге” и т.д.

---

### 3) Встраивание в пайплайн

Высокоуровневый поток:

1. Подготовка аудио (`core/ffmpeg.py`).
2. ASR + alignment на уровне слов (WhisperX) → word timestamps.
3. DisfluencyDetection (новый шаг):
   - FillerWordsDetector (словарь + alignment)
   - FillerSoundsDetector (токены + акустические эвристики)
   - BreathsDetector (эвристика + опционально PANNs‑классификатор)
4. Пост‑обработка интервалов: слияние, границы, лимиты, конфиденс.
5. Формирование cutlist JSON.
6. Применение: Attenuate (по умолчанию) или Cut+Crossfade (опция) к WAV‑копии.
7. Экспорт: обновлённые SRT/VTT, EDL для NLE, JSON‑отчёт метрик.

Позиция в коде: опциональный шаг в `core/pipelines.LocalPipeline` после Alignment и до Exporters.

---

### 4) Методы детекции

#### 4.1 Слова‑паразиты (ru)

- База: словарь устойчивых маркеров (“ну”, “типа”, “как бы”, …). Списки расширяемы в настройках.
- Привязка к аудио: word‑timestamps из WhisperX.
- Нормализация: простая лемматизация для устойчивых выражений (опционально). Для MVP достаточно lowercased exact‑match на токены WhisperX.
- Защита от ложных срабатываний:
  - Режимы strict/normal/aggressive.
  - Белый список слов (“да” в ответах) и исключения по контексту.
  - Лимит доли удаления на фразу/минуту (например ≤10%).

#### 4.2 Звуки‑филлеры (“э‑э”, “мм”)

- Текстовый канал: последовательности токенов типа “э”, “эээ”, “м”, “ммм” из транскрипта.
- Акустические эвристики для пропусков ASR:
  - Устойчивая гласная с низкой спектральной динамикой (малый spectral flux), невысокая формантная вариативность.
  - Длительность 80–400 мс, энергия выше шумового порога, отсутствует чёткая согласная атака.
- Слияние близких интервалов, минимальная/максимальная длительность, confidence по сочетанию каналов.

#### 4.3 Вздохи/дыхание

- Эвристика:
  - Рассматривать не‑речевые участки (по VAD/ASR‑паузы).
  - Признаки: spectral slope, соотношение полос (например, 300–1500 Гц), огибающая, ZCR.
  - Длительность 200–1200 мс, отступы от соседних слов.
- Опционально PANNs (AudioSet‑класс “Breathing/Sigh”) для дополнительной уверенности.
- По умолчанию действие: attenuate (−12 dB, линейная/косинусная огибающая 10–30 мс).

---

### 5) Применение правок (безопасность)

- Attenuate (дефолт):
  - Gain: −12 dB (настраиваемо), fade‑in/out 10–30 мс, тип огибающей — косинус/экспонента.
  - Практически не ломает ритм, минимальные артефакты.
- Cut + Crossfade (опция):
  - Выбор ripple/non‑ripple:
    - Non‑ripple: оставляем таймлайн неизменным, заменяя вырез тишиной/приглушением.
    - Ripple: время сдвигается; подходит для аудио‑подкаст версии, но требует внимательности к субтитрам/видео.
  - Crossfade 2–40 мс (адаптивно по скорости речи и окружению).
- Ограничения:
  - Максимум удалений на единицу речи (защита от “скачущей” фразы).
  - Не вырезать если интервал перекрывает слово >30–40% (переключаться в attenuate).

### 6) Формат cutlist JSON

```json
{
  "version": "1.0",
  "source": {
    "file": "PATH_TO_INPUT",
    "sample_rate": 16000
  },
  "items": [
    {
      "id": "cl_00001",
      "type": "filler_word",  
      "start_ms": 1250,
      "end_ms": 1420,
      "confidence": 0.93,
      "action": "attenuate",  
      "margin_before_ms": 20,
      "margin_after_ms": 20,
      "tags": ["ru_dict:ну"]
    }
  ]
}
```

Типы: `filler_word` | `filler_sound` | `sigh`.
Action: `attenuate` | `cut`.

---

### 7) Настройки (settings.json)

- cleanup.enable: true|false
- cleanup.types: { filler_words: true, filler_sounds: true, breaths: true }
- cleanup.strategy: "attenuate" | "cut"
- cleanup.limits: { per_sentence_max_fraction: 0.1, per_minute_max_ms: 8000 }
- cleanup.margins_ms: { before: 20, after: 20 }
- cleanup.fade_ms: { in: 15, out: 15 }
- cleanup.gain_db: -12
- cleanup.filler_words.dict: ["ну", "типа", "как бы", ...]
- cleanup.thresholds: { sigh_conf: 0.6, sound_conf: 0.6 }

Все значения — переопределяемые через GUI/CLI.

---

### 8) CLI

```bash
speach-kit transcribe \
  --input PATH_TO_MEDIA \
  --cleanup-disfluencies \
  --cleanup-types filler_words,filler_sounds,breaths \
  --cleanup-strategy attenuate \
  --cleanup-gain-db -12 \
  --cleanup-margins 20,20 \
  --export edl,json,srt
```

### 10) Оценка качества и автоматика

- Авто‑разметка: генерируются позитивные примеры из транскрипта (словарь) и эвристик (вздохи/звуки). Нужен небольшой ручной аудит 30–60 минут только для низкоуверенных случаев.
- Метрики: precision/recall per‑type, суммарная доля правок, число артефактов (счётчик manual‑revert в GUI).
- CI‑гейты: билд падает, если метрики падают ниже порога.

---

### 11) Датасеты и источники

- Русские корпуса речи без дисфлюенций: `OpenSTT`, `Golos`, `Common Voice (ru)` — подходят для акустических калибровок/порогов.
- Универсальные аудио‑классы: предобученные PANNs по `AudioSet` с классами breathing/sigh — годится для валидации дыхания без обучения.
- Собственная авто‑разметка по выравненному тексту обеспечивает старт без ручной аннотации.

Примечание: соблюдайте лицензии датасетов. Секреты (например, токены) хранить в переменных окружения (`YOUR_HF_TOKEN`).

### 12) Альтернативные варианты

1) Плагины DAW (iZotope RX Breath Control, Waves DeBreath):
   - Плюсы: качественная обработка дыхания из коробки.
   - Минусы: проприетарно, нет правок “по словам”, сложнее автоматизировать в нашем пайплайне.

2) Контекстная модель дисфлюенций (sequence tagging, RuBERT‑класс):

   - Плюсы: лучше различает семантические случаи (“да” как ответ vs дисфлюенция).
   - Минусы: требует обучающего набора и инфра для инференса; позже как Stage 4.

3) YAMNet вместо PANNs:
   - Плюсы: готовые эмбеддинги/классы.
   - Минусы: зависимость от TensorFlow; проект сейчас в Torch‑экосистеме — не выбрано.

4) Облачные редакторы (Descript‑подобные):
   - Плюсы: готовые “Remove filler words”.
   - Минусы: облако, часто EN‑фокус, нет офлайн‑гарантий и открытого API под наши нужды.

5) Спектральная маскировка вместо выреза (динамический EQ/спектральный гейт):
   - Плюсы: минимум склеек, подходит для звуков‑филлеров и дыхания поверх слова.
   - Минусы: возможна деградация естественности тембра; оставляет шумовой след.

---

### 13) Риски и меры

- Склейки/щелчки при вырезах → дефолт: attenuate; при cut — короткие кросс‑фейды.
- Семантические слова (“да/вот/ну”) → режим strict, белые списки, метрики откатов в GUI.
- Сложные шумовые окружения → адаптивные пороги, fallback в attenuate.
- Производительность → CPU‑совместимо; PANNs включать опционально; батч‑размеры управляемы.

---

### 14) План внедрения (этапы)

- Stage 1 (MVP):
  - FillerWordsDetector по словарю + alignment; Cutlist JSON; действие: attenuate.
  - CLI/GUI флаги; базовые метрики.

- Stage 2:
  - FillerSoundsDetector: токены + акустические эвристики; EDL экспорт; предпросмотр в GUI.

- Stage 3:
  - BreathsDetector: эвристика + опционально PANNs; тонкая настройка порогов.

- Stage 4:
  - Контекстная модель дисфлюенций (RuBERT), white/black‑lists, улучшение метрик.

### 16) Безопасность и секреты

- Не хранить ключи/токены в коде. Использовать переменные окружения (`YOUR_HF_TOKEN`, `YOUR_MODEL_PATH`).

### 17) Примечания по производительности

- Все этапы работают на CPU; ускорение на GPU опционально.
- PANNs включать только по запросу пользователя.
- Обрабатывать большими окнами с шагом 10–20 мс; batch‑ориентировать, где возможно.
