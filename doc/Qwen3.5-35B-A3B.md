# Qwen3.5-35B-A3B for AskVLM

Краткая внутренняя карточка по целевой локальной модели для `Video QA` и смежных мультимодальных сценариев.

## 1. Что это за модель

- Модель: `Qwen/Qwen3.5-35B-A3B`
- Класс: мультимодальная LLM (`text + image`)
- Архитектура: MoE, всего `35B`, активируется около `3B` параметров на токен
- Практический смысл для проекта: одна сильная локальная модель для русского языка, reasoning и image-conditioned ответов через LM Studio

## 2. Почему она подходит AskVLM

- Поддерживает мультиязычные сценарии, включая русский.
- Подходит для grounded-ответов по видео, если в prompt передаются выдержки транскрипта и representative frames.
- Даёт достаточно сильный reasoning-слой, чтобы собирать итоговый ответ по чанкам, а не только описывать кадры.
- Формально имеет большой контекст, но в приложении всё равно нужен строгий budget и chunking.

## 3. Ограничения, которые нельзя игнорировать

- MoE-модель на локальном железе сильно зависит от квантизации и качества конкретной сборки в LM Studio.
- Наличие vision в названии/карточке не гарантирует, что конкретный локальный артефакт действительно принимает изображения в нужном формате.
- Structured output может быть нестабильным в локальном OpenAI-compatible стеке; нужен fallback через валидацию и retry.
- Большой контекст не отменяет ограничений по RAM, latency и качеству ответа на длинных мультимодальных входах.

## 4. Практические ориентиры для AskVLM

- Держать `Qwen3.5-35B-A3B` как primary candidate для `Video QA`, а не как безусловно зафиксированный production standard.
- Использовать её после preflight по budget: текст считать точнее, изображения оценивать эвристикой с запасом.
- Не запускать параллельно с другой тяжёлой моделью без scheduler'а.
- Проверять отдельно три сценария:
  - текстовый вопрос без изображений;
  - вопрос по транскрипту плюс несколько кадров;
  - длинный вопрос с вложениями и evidence-ориентированным JSON-ответом.

## 5. LM Studio caveats

- Работать через OpenAI-compatible endpoint, но не предполагать полную идентичность облачным провайдерам.
- До включения structured output в основной pipeline проверить, как модель ведёт себя на реальном JSON-контракте AskVLM.
- До включения production path проверить фактическое поведение при переполнении контекста: ошибка, partial output, `stopReason` или иной сигнал.
- Если vision-path в конкретной сборке нестабилен, оставить модель как текстовый aggregator и временно ослабить мультимодальный проход.

## 6. Рабочее решение на сейчас

- Да, хранить карточку этой модели в репозитории полезно.
- Эта модель остаётся целевым кандидатом для первого прохода `Video QA` через LM Studio.
- Выбор конкретной квантизации и окончательная фиксация в коде делаются только после живой проверки мультимодального сценария на целевом железе.

## 7. Что ещё нужно проверить руками

- Какой именно артефакт загружается в LM Studio: текстовый или мультимодальный.
- Какой реальный usable context получается на целевом железе.
- Даёт ли модель устойчивый JSON-ответ с evidence-полями.

## 9. LM Studio — internal reference (verified from official docs)

Краткая зафиксированная семантика для backend (`Video QA`), без замены живой проверки на железе.

- **OpenAI-compatible HTTP**: в документации LM Studio описаны OpenAI-совместимые маршруты, включая `/v1/responses` и `/v1/chat/completions`. Конкретная форма JSON и наличие полей зависят от endpoint и версии; нельзя считать, что termination-метаданные всегда идентичны облачному OpenAI.
- **Multimodal payload**: запросы с изображениями следуют OpenAI-совместимому формату (например, `content` как список частей с `image_url` / base64); детали и ограничения — в официальных разделах LM Studio про OpenAI compat и image input.
- **Streaming**: при SSE/streaming-режиме итоговые `finish_reason` / `stopReason` могут появляться только на последнем chunk или в отдельной структуре; клиенту нужно дождаться завершения потока, прежде чем полагаться на нормализацию termination.
- **`stopReason` (prediction stats)**: документированные значения включают `maxPredictedTokensReached`, `stopStringFound`, `toolCalls`, `contextLengthReached`. Код нормализации: `core/video_qa_lm_studio_termination.py`.
- **`contextOverflowPolicy`**: поддерживаемые режимы — `stopAtLimit`, `truncateMiddle`, `rollingWindow`. Это server-side политика относительно контекста; приложение по-прежнему опирается на собственный preflight и fallback до отдельной верификации на целевой модели.
- **Structured output**: совместимость с жёстким JSON/схемой зависит от модели и сборки; graceful fallback остаётся обязательным.

Ограничение: end-to-end поведение при переполнении контекста (ошибка vs partial output vs тихая усечённость) на целевом железе и версии LM Studio остаётся **ручной** проверкой; этот раздел не отменяет пункт roadmap про live overflow verification.
